FIDELIZATE · Configuración de GHL

Guía para configurar campos personalizados y workflows en cada subcuenta. Hecha para copiar/pegar rápido.

0 Idea base: dos dimensiones

Progreso

En qué sello va el cliente. La calcula FIDELIZATE y la manda en cada compra. Sirve para "te falta 1", "premio listo".

Recencia

Cuántos días lleva sin comprar. La maneja GHL con temporizadores anclados a la fecha de última compra, que se reinician en cada compra.

FIDELIZATE manda valores absolutos (no "suma 1"). GHL solo guarda y reacciona. Reenviar el mismo evento nunca rompe nada.

1 Campos personalizados a crear

En la subcuenta: Settings → Custom Fields → Add Field. Crea estos (objeto: Contact).

NombreTipoPara qué
loyalty_stampsNumberSellos en la tarjeta actual (0–10)
loyalty_totalNumberCompras históricas totales
loyalty_to_rewardNumberSellos que faltan para el premio
loyalty_reward_readyCheckboxTarjeta completada (premio disponible)
loyalty_last_purchaseDateFecha de última compra (ancla de recencia)
loyalty_stageSingle line / DropdownEtapa de progreso (NUEVO, EN_PROGRESO, CASI_PREMIO, PREMIO_LISTO)
rutSingle lineRUT del cliente (identificador)

2 Qué envía FIDELIZATE (payload) y a qué campo va

{ "rut": "18.431.296-4", → contacto: custom field rut + dedup "name": "Mauricio", → contacto: Name "email": "cliente@correo.cl", → contacto: Email "phone": "+56912345678", → contacto: Phone (llave de dedup) "stampCount": 1, → loyalty_stamps "previousStampCount": 0, → (para mensajes "te falta 1") "stampsToReward": 9, → loyalty_to_reward "totalPurchases": 1, → loyalty_total "lastPurchaseAt": "2026-...Z", → loyalty_last_purchase "rewardAt": 10, → (referencia) "rewardTriggered": false, → rama premio + loyalty_reward_ready "isNewCustomer": true, → rama bienvenida "loyaltyStage": "NUEVO", → loyalty_stage "ghlContactId": "" → (referencia, puede venir vacío) }
En el workflow, estos valores se eligen del selector de variables del webhook. Aparecen como {{inboundWebhookRequest.stampCount}}, {{inboundWebhookRequest.loyaltyStage}}, etc.

3 Workflow 1 · "FIDELIZATE · Actualización de datos"

Trigger: el Inbound Webhook (la compra). Procesa datos y manda solo 2 mensajes pagados: bienvenida (1ª) y premio.

① Trigger · Inbound Webhook
El que ya creaste (.../webhook-trigger/8ed88bf8...). Reference payload mapeado.
② Create / Update Contact (Upsert)
Dedup por Phone. Registra/identifica el número.
Phone = {{inboundWebhookRequest.phone}} Email = {{inboundWebhookRequest.email}} Name = {{inboundWebhookRequest.name}} rut = {{inboundWebhookRequest.rut}}
③ Update Contact Field (5 campos)
loyalty_stamps = {{inboundWebhookRequest.stampCount}} loyalty_total = {{inboundWebhookRequest.totalPurchases}} loyalty_to_reward = {{inboundWebhookRequest.stampsToReward}} loyalty_last_purchase = Current Date loyalty_stage = {{inboundWebhookRequest.loyaltyStage}}
④ If isNewCustomer = true → mensaje bienvenida
Solo en la 1ª compra. (Mensaje pagado #1.)
⑤ If rewardTriggered = true → premio
Set loyalty_reward_ready=true · WhatsApp "tu premio está listo" · Add Tag loyalty_reward_pending. (Mensaje pagado #2, ~1 cada 10 compras.)
⑥ Reinicia el seguimiento
Add to Workflow → "FIDELIZATE · Seguimientos". Eso saca al cliente del ciclo viejo de no-compra y reinicia el reloj (ver Workflow 2).
La etapa vive en el campo loyalty_stage (valor único que se sobrescribe), así que NO se usan etiquetas de etapa: para filtrar o condicionar usa directo el valor del campo. El único tag es loyalty_reward_pending (premio sin canjear). Y NO hay mensajes por compra intermedia: actualizar campos es gratis, lo que cuesta es enviar.

4 Workflow 2 · "FIDELIZATE · Seguimientos" (no-compra)

Los recordatorios por inactividad. Una compra (Workflow 1) lo reinicia.

Settings del workflow: Allow Re-Entry: ON + Remove from this workflow on re-entry: ON. Trigger: se entra por el "Add to Workflow" del paso ⑦ (puedes dejar el trigger en un tag tipo cliente-activo o entrada manual; el alta real la hace Workflow 1).

Wait 15 días → recordatorio + progreso
"Te faltan {{contact.loyalty_to_reward}} sellos para tu premio."
Wait hasta 30 días → "te extrañamos"
Wait hasta 60 días → descuento de reactivación
(opcional) 90 días → tag lealtad-perdido
Punto crítico: el reinicio se hace re-agregando a ESTE workflow (con Re-Entry on, GHL saca la instancia anterior y arranca de cero). Nunca uses "remove from ALL workflows" — eso apagaría también tus campañas de publicidad mensual. Al nombrar solo "FIDELIZATE · Seguimientos", la publicidad mensual (otros workflows) queda intacta.

60d: "Tenemos algo para ti: [DESCUENTO]. Válido en tu próxima visita."

5b Workflow 3 · "FIDELIZATE · Entrega de premio" (canje)

Trigger: un 2º Inbound Webhook (distinto al de compra). Lo dispara el botón "Canjear premio" de la caja.

Payload que envía FIDELIZATE al canjear:

{ "event": "reward_redeemed", "rut": "18.431.296-4", "name": "Mauricio", "phone": "+56983971952", "rewardsRedeemed": 1, "redeemedAt": "2026-...Z" }
① Trigger · Inbound Webhook (canje)
URL .../webhook-trigger/a08a0a00... Reference payload mapeado.
② Find/Update Contact
Dedup por {{inboundWebhookRequest.phone}}.
③ Remove Tag
loyalty_reward_pending · y Set loyalty_reward_ready = false
④ (opcional) Mensaje "gracias por canjear"
"¡Disfruta tu premio! Empieza tu nueva tarjeta."
El estado real del canje vive en FIDELIZATE (rewardPending, rewardsRedeemed). Este workflow solo limpia el tag en GHL y agradece. El cajero confirma el canje físico con el botón de la caja.

6 Orden para configurar rápido

1. Crear los 7 campos personalizados (sección 1).

2. Crear el Workflow 2 "FIDELIZATE · Seguimientos" (Re-Entry + remove-on-re-entry ON) con los Wait 15/30/60/90.

3. Crear el Workflow 1 "FIDELIZATE · Actualización de datos" (sección 3); en el paso ⑦, Add to Workflow → Seguimientos.

4. Publicar ambos.

5. Desde FIDELIZATE → caja, registrar una compra de prueba y ver el contacto actualizarse + entrar a Seguimientos.

Re-selecciona el reference payload del Inbound Webhook si apunta a un ejemplo viejo: manda una compra de prueba, "Check for new requests", y elige el request que ya trae loyaltyStage / isNewCustomer.
